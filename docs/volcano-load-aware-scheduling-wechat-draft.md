# 让可观测数据进入调度闭环：Volcano 负载感知调度增强实践

在云原生调度系统中，“当前最空闲的节点”并不总是“下一批 Pod 最适合落下去的节点”。

这个判断听起来有些反直觉。调度器已经接入了节点 CPU、内存利用率，为什么还会把 Pod 调度到即将变热的节点上？问题的关键在于：监控系统看到的是已经采集、聚合并上报后的负载，而调度器正在面对的是一个连续发生的决策过程。当多个 Pod 在短时间内批量创建或重部署时，刚刚调度出去的新 Pod 尚未体现在节点利用率指标中，调度器如果只看外部指标，就容易反复选择“当前打分最优”的节点，最终制造新的热点。

这次 Volcano 负载感知调度增强，解决的就是这个监控延迟窗口带来的调度偏差。它让调度器不仅使用外部可观测数据，也把自己刚刚做出的调度决策纳入当前负载视图，形成一个更接近实时状态的调度闭环。

## 真实客户需求

这个需求来自真实客户场景，而不是抽象出来的功能优化。

人人视频客户在将主站业务从阿里云迁移到华为云时，在华为云 CCE 集群中使用 Volcano 负载感知调度插件。迁移过程中，客户发现一个高频问题：在批量新建 Pod 或重部署 Pod 时，Volcano 会大量将 Pod 调度到当前打分最优的节点上。短时间内看，这个节点的 CPU、内存指标仍然较低，调度分数也较高；但随着新 Pod 启动并开始产生真实资源消耗，这个节点很快可能成为新的负载热点。

客户反馈，阿里云侧调度插件具备类似能力：在计算节点打分时，如果节点上存在“利用率还未上报的新 Pod”，会对打分结果做适当调整，从而避免过量调度到同一个节点。华为云 CCE 当前 Volcano 插件缺少这类能力，导致客户在主站业务迁移和日常重部署场景中反复遇到热点问题。

从需求影响看，这不是低频边缘问题：

- 紧急程度：高；
- 影响业务频次：每天都在发生；
- 影响用户范围：主要影响运维人员和负责业务迁移、发布、扩缩容的集群管理者；
- 典型场景：批量新建 Pod、滚动重部署、节点扩容后回填任务、业务高峰前集中发布。

因此，这个需求的目标不是单纯“让调度更平均”，而是在负载感知调度已有能力基础上，补齐新 Pod 尚未进入监控指标之前的判断能力，避免调度器在短时间内持续低估目标节点压力。

## 背景导入：上一篇解决了什么，这一篇继续解决什么

Volcano 原有 usage 插件已经解决了一个关键问题：调度器不再只看 Pod request、limit 和节点 allocatable，而是可以接入 Prometheus、Custom Metrics API、Elasticsearch 等外部指标源，在调度时参考节点真实 CPU、内存利用率。

这个能力主要体现在两个阶段。

第一，在 Predicate 阶段，usage 插件可以根据用户配置的 CPU、内存阈值过滤高负载节点。如果节点真实 CPU 或内存利用率已经超过阈值，就不再继续向该节点调度新 Pod。

第二，在 NodeOrder 阶段，usage 插件会根据节点负载进行打分。负载越低，得分越高；节点越忙，得分越低。这样，调度器可以优先选择运行时负载更低的节点。

上一篇负载感知调度文章关注的就是这件事：调度器如何“看见”节点真实负载，并基于负载参与调度决策。

但真实客户场景暴露出下一层问题：调度器看见的负载并不一定等于调度瞬间的真实负载。外部指标有采集周期，调度器也有拉取周期。一个 Pod 刚刚被调度到节点后，调度器已经知道这个事件发生了，但 Prometheus 或 Custom Metrics API 还没有来得及把这个 Pod 的资源消耗反映到节点曲线上。

在单个 Pod 调度场景下，这个时间差影响不明显。但在批量新建、滚动重部署、Gang 调度等场景中，多个 Pod 会在一个或几个调度周期内连续完成节点选择。此时，如果某个节点在外部指标里仍然“看起来很空”，调度器可能反复选择它。等下一轮指标上报时，热点已经形成。

这就是本文要解决的问题：在已有 usage-based scheduling 的基础上，让调度器把“已经调度但尚未被监控系统看见的新 Pod”也纳入节点负载判断。

换句话说，上一阶段解决“如何基于真实负载调度”，这一阶段解决“如何在监控延迟窗口内仍然尽量接近真实负载”。

## 技术选型：为什么选择节点视角的影子负载方案

在设计这个特性时，有几个可选方向。

一种思路是从 Pod 视角出发：根据每个 Pod 的属性、业务类型、历史画像、实际资源使用情况，判断它更适合调度到哪个节点。这个方案看起来更精细，但落地成本很高。首先，新 Pod 刚启动时没有实际利用率数据；其次，不同业务的历史画像不一定可得，也不一定稳定；再次，如果调度逻辑强依赖每个 Pod 的业务语义，就会把通用调度插件变成强业务绑定的预测系统。对 Volcano 这类通用调度器来说，这会显著增加复杂度和维护成本。

另一种思路是对节点打分做固定惩罚：只要节点在当前周期内分配过新 Pod，就扣一定分数。这种方案实现简单，但准确性不足。不同 Pod 的 request、limit、QoS、资源维度差异很大。一个 50m CPU 的小 Pod 和一个 4 核 CPU 的大 Pod，不应该对节点分数产生同样影响。固定惩罚要么过于保守，影响集群利用率；要么过于激进，无法真正规避热点。

还有一种思路是只调整扩散策略，例如强制 Pod 更均匀地分散到不同节点。这对部分场景有效，但它不是负载感知。节点之间容量不同、当前负载不同、CPU/Mem 压力不同，简单分散并不能代表更优调度。客户问题的本质不是“Pod 数量是否平均”，而是“新增负载是否被当前节点分数正确反映”。

最终选择的是节点视角的影子负载方案。

这个方案保留了 usage 插件原有设计：调度器仍然以节点为打分对象，仍然使用 CPU/Mem 真实利用率，仍然在 Predicate 和 NodeOrder 中完成决策。新增的只是一个调度器内部可维护的补充视图：当某个 Pod 已经被调度到节点、但其资源消耗尚未进入外部指标时，调度器先为它估算一份资源占用，把这份估算临时计入节点负载。后续节点打分时，使用真实负载加影子负载共同计算。

选择这个方案的原因有三点：

第一，它与现有 usage 插件模型一致。原有插件本来就是节点视角，CPU/Mem 指标也是节点维度，新增影子负载只是在同一个维度补齐监控延迟窗口。

第二，它对业务无侵入。不需要客户给每类业务维护画像，也不要求业务改造 Pod 模板，只通过已有 request、limit、QoS 和可配置参数进行估算。

第三，它能兼顾实时性与可解释性。调度器知道自己刚分配了哪些 Pod，也知道每个 Pod 的资源声明；这些信息比外部监控更早到达调度路径，同时又比固定惩罚更接近真实资源影响。

## 怎么做：核心原理

这次增强由三部分组成：资源预估机制、影子负载机制、以及 Pod 资源预估的新增与回退机制。

### 资源预估机制

对 Guaranteed 和 Burstable 类型的 Pod，插件使用 request、limit 和可配置权重估算资源消耗：

```text
pod_estimate = (request * request_ratio + (limit - request) * burst_ratio) * applied_risk_factor
```

`request_ratio` 表示 request 部分对估算值的贡献，`burst_ratio` 表示 request 到 limit 之间弹性空间的贡献。默认配置下，request 是主要参考，burst 部分默认不计入；如果客户业务突发性更强，可以提高 `burst_ratio`。

估算值会被限制在合理范围内：不会低于 0，也不会超过有效 limit。如果某个资源没有 limit，或者 limit 小于 request，则以 request 作为有效上限。这样可以避免对无 limit Pod 做过度夸大的估算，也避免异常配置影响节点打分。

对 BestEffort Pod，由于没有 request 和 limit，插件提供固定估算值配置，例如默认 CPU 为 `250m`，内存为 `200Mi`。这部分配置可以结合客户环境调优，避免 BestEffort Pod 被完全低估。

### 风险系数机制

仅估算单个 Pod 还不够。节点当前越忙，同样一个新 Pod 带来的风险越高。因此插件引入了 `risk_threshold` 和 `risk_factor`。

调度器会先计算节点复合利用率：

```text
cpu_composite = (real_cpu_load + shadow_cpu_estimate) / node_cpu_capacity
memory_composite = (real_memory_load + shadow_memory_estimate) / node_memory_capacity
load_composite = weighted_average(cpu_composite, memory_composite)
```

当 `load_composite` 达到 `risk_threshold` 后，后续新增 Pod 的估算值会乘以 `risk_factor`。这相当于告诉调度器：节点已经接近风险水位，不要再用普通估算值低估新 Pod 对节点的影响。

### 影子负载机制

影子负载是 session 级别的节点估算负载缓存。每个节点维护两类累计值：CPU 影子负载和内存影子负载。每个被计入缓存的 Pod 也会保存一份快照，记录它当时被估算为多少 CPU、多少内存、落在哪个节点。

NodeOrder 打分时，节点使用的不是单纯外部指标：

```text
real_load
```

而是组合后的负载：

```text
real_load + shadow_load
```

这样，在外部指标尚未上报之前，调度器也能意识到某个节点刚刚已经被放入了一批新 Pod，从而降低它在后续调度中的分数。

### 新增与回退机制

当 Pod 在当前调度 session 中被 Allocate 到某个节点时，usage 插件会估算它的 CPU/Mem 消耗，并将估算值加入该节点的影子负载。

当调度过程发生 Deallocate，也就是调度回滚时，插件不会重新计算一遍估算值，而是使用 Allocate 时保存的快照做精确扣减。这一点对 Gang 调度尤其重要：如果一组 Pod 不能整体调度成功，前面已经临时分配出去的 Pod 需要回滚。影子负载如果不能同步回退，后续调度就会错误地高估节点负载。

因此，这次实现采用“加的时候记录快照，减的时候按快照扣回”的策略，保证影子负载缓存和调度事件保持一致。

## 怎么实操：工程实现中的关键处理

### 如何判断节点是否过热

节点是否过热分成两个层次。

第一层是硬阈值判断。Predicate 阶段仍然只看真实监控指标。如果节点真实 CPU 或内存利用率超过用户配置的 `thresholds.cpu` 或 `thresholds.mem`，节点会被过滤。这保证已经明确过载的节点不会继续接收新 Pod。

第二层是排序阶段的风险判断。NodeOrder 阶段会把真实负载和影子负载合并成复合利用率，再计算节点得分。复合利用率越高，节点得分越低。当复合利用率超过 `risk_threshold` 后，新增 Pod 的估算还会乘以 `risk_factor`，让后续调度更保守。

这样设计的好处是，硬过滤仍然基于真实观测，避免估算导致误杀；排序阶段则使用影子负载弥补监控延迟，避免持续把新 Pod 调度到同一个“看起来还很空”的节点。

### 为什么 Pod 预估值要这样设计

Pod 预估值没有直接使用 limit，也没有完全使用 request，而是提供 `request_ratio` 和 `burst_ratio` 两个参数。

原因是客户业务差异很大。对资源声明比较准确的业务，request 往往已经能较好反映稳定消耗；对突发性强的业务，仅看 request 又容易低估启动期或高峰期压力。通过两个参数组合，用户可以在“更保守”和“更高利用率”之间调整。

默认配置偏向稳妥：以 request 为主要依据，避免所有 Pod 都按 limit 估算导致过度分散。对于 BestEffort Pod，由于没有 request/limit，则通过 `be_cpu` 和 `be_memory` 给出可配置的固定估算值，避免 BestEffort 完全不影响节点打分。

实现上还处理了几个边界：

- request 为负数时按 0 处理；
- limit 缺失或小于 request 时，以 request 作为有效 limit；
- 估算结果被限制在 `[0, effectiveLimit]` 范围内；
- initContainers 按 Kubernetes 资源语义取最大值，而普通 containers 按总和计算；
- `risk_factor` 小于 1 时不会降低估算值，避免在高压场景下反而更激进。

这些处理保证了估算值可配置，但不会因为异常配置导致明显不合理的节点分数。

### 如何处理 Gang 调度和回滚

Volcano 面向批处理和 AI 任务，一个典型能力是 Gang 调度：一组 Pod 要么整体调度成功，要么整体等待。这个场景下，调度器可能先临时为部分 Pod 选择了节点，但最终发现整个 PodGroup 无法满足条件，需要回滚。

如果影子负载只在 Allocate 时增加，不在 Deallocate 时精确回退，就会出现两个问题：

第一，节点负载被高估。已经回滚的 Pod 并不会真的运行，但影子负载还留在缓存中。

第二，后续调度被污染。调度器会认为某些节点已经承载了不存在的负载，从而错误避开这些节点。

为避免这个问题，ShadowLoadCache 为每个 Pod 保存 `PodEstSnapshot`。快照里记录节点名、CPU 估算值、内存估算值。Deallocate 时直接按快照扣减，不依赖重新计算。即使节点压力、风险系数或配置在加减之间发生变化，也不会影响回滚一致性。

### 如何保证影子负载缓存不低估也不高估

影子负载最重要的是正确性。它既不能低估节点压力，否则热点问题仍然存在；也不能长期高估节点压力，否则会牺牲集群利用率。

这次实现通过几个机制保证正确性。

首先，session 打开时会初始化 ShadowLoadCache，并读取 `metrics.interval` 作为监控延迟窗口。插件会扫描当前节点上已有任务，把已经分配到节点、但可能尚未被指标系统覆盖的 Pod 加入缓存。状态包括 Allocated、Binding、Bound，以及启动时间短于 metrics interval 的 Running Pod。

其次，Pending、Succeeded、Failed 等状态不会进入影子负载。已经完成或尚未绑定节点的任务，不应该影响节点估算负载。

再次，缓存支持同一个 task 的更新。如果同一个 Pod 的估算被重新加入，旧快照会先被扣除，再加入新快照，避免重复计算。

最后，ShadowLoadCache 使用锁保护读写，保证并发访问下不会出现负值或不一致；session 关闭时也会清理缓存，避免跨 session 残留。

### 为什么不把影子负载用于硬过滤

这次实现中，Predicate 仍然只检查真实指标是否超过阈值，影子负载主要用于 NodeOrder 排序。这是一个有意的工程取舍。

影子负载本质上是估算值。估算值适合影响优先级，但如果直接用于硬过滤，可能在配置偏保守时过早排除节点，影响任务可调度性。尤其在 Gang 调度场景下，过度过滤可能放大调度失败概率。

因此，方案把“不能调度”的判断留给真实指标，把“更不应该优先调度”的判断交给组合负载。这既保留了负载感知调度的安全边界，也让增强能力更平滑地作用在节点排序上。

## 配置示例

这次增强延续 usage 插件已有配置方式，同时新增 estimator 相关参数：

```yaml
actions: "enqueue, allocate, backfill"
tiers:
  - plugins:
      - name: usage
        enablePredicate: false
        arguments:
          usage.weight: 5
          cpu.weight: 1
          memory.weight: 1
          thresholds:
            cpu: 80
            mem: 80
          estimator:
            request_ratio: 0.7
            burst_ratio: 0
            risk_threshold: 0.6
            risk_factor: 1.2
            be_cpu: 250m
            be_mem: 200Mi
metrics:
  type: prometheus
  address: http://prometheus:9090
  interval: 30s
```

其中，`metrics.interval` 不只影响指标拉取频率，也用于判断监控盲区。对刚启动不久的 Running Pod，如果运行时间尚未覆盖这个 interval，就会被认为仍可能没有进入外部指标。

参数调优可以按业务类型区分：

- 稳态业务：提高 `request_ratio`，保持估算接近资源声明；
- 突发业务：适当提高 `burst_ratio`，降低 request 偏低导致的低估；
- 高风险业务：降低 `risk_threshold` 或提高 `risk_factor`，让节点接近高水位时更保守；
- BestEffort 较多的集群：结合历史运行数据校准 `be_cpu` 和 `be_memory`。

## 痛点、挑战、方案、亮点

可以把这个特性的价值总结为四个关键词。

痛点是监控延迟。客户不是没有负载感知能力，而是在批量调度场景中，真实指标来得比调度决策慢。当前看起来最优的节点，可能因为连续接收新 Pod 而很快成为热点。

挑战是调度正确性。调度器需要提前考虑新 Pod 的影响，但又不能过度依赖估算值；需要在 Gang 调度回滚时精确扣减，又不能让影子负载跨周期污染后续调度；需要给客户足够大的可配置空间，又要避免配置异常导致明显错误的估算。

方案是节点视角的影子负载。它沿用 usage 插件的节点打分模型，使用资源预估机制计算新 Pod 对节点 CPU/Mem 的影响，再将估算值加入 session 级 ShadowLoadCache。后续 NodeOrder 使用真实负载和影子负载共同计算分数，Deallocate 时按快照精确回退。

亮点是低侵入、可解释、可回退。业务不需要改造，调度器不需要依赖复杂任务画像；每个节点为什么降分可以从真实负载、影子负载、风险系数中解释；每次临时调度失败后，影子负载都能按快照恢复，保证调度周期内的一致性。

## 方案选型与友商对比

客户反馈中提到，阿里云侧调度插件在节点打分时会考虑“利用率尚未上报的新 Pod”，从而避免过量调度产生热点。这个反馈直接说明：客户关注的不是单纯节点分散，而是调度器是否能理解监控延迟窗口中的新增负载。

在增强后，我们收集客户真实生产环境关键信息，并模拟客户真实环境进行验证。同时，在友商 ACK 集群上构建相同环境进行横向对比。验证结论可以概括为：大多数场景下，增强后的 Volcano 负载感知调度表现与友商方案持平，部分场景略占优；在少数场景中，Koordinator 当前实现无法覆盖，会出现 OOM 风险，而增强后的 Volcano 方案可以通过影子负载和资源预估提前降低热点节点得分，规避继续集中调度。

[待补充：这里建议加入测试报告中的具体场景、Pod 数量、节点规格、重部署批次、ACK/CCE 对比截图、节点负载曲线、OOM 复现条件和最终结果表。]

从方案选型上看，这次增强并不是复制某个友商能力，而是在 Volcano usage 插件现有架构下做了更自然的扩展：真实指标仍来自 metrics source，硬阈值仍在 Predicate 生效，调度排序仍在 NodeOrder 完成；新增的影子负载只负责补齐“调度器已知但监控尚未知”的信息。

这种实现方式更适合 Volcano 的批处理和 Gang 调度场景。因为它不仅要处理单个 Pod 的连续调度，还要处理 PodGroup 整体调度失败后的回滚一致性。快照式扣减正是为这类场景设计的。

## 落地效果如何验证

针对这类调度增强，验证不能只看单次调度结果，而要看连续调度过程中的节点负载走势。

建议从几个维度验证：

第一，批量新建场景。连续创建一批 Pod，观察增强前后 Pod 是否仍集中落到最初打分最优的节点，以及后续节点分数是否随影子负载变化而下降。

第二，重部署场景。模拟客户日常发布，观察滚动重建过程中是否出现局部节点负载快速升高，以及增强后热点是否被削弱。

第三，Gang 调度场景。构造部分 Pod 临时分配后整体调度失败的 PodGroup，验证 Deallocate 后影子负载是否准确回退。

第四，BestEffort 或 request 偏低场景。验证 `be_cpu`、`be_mem`、`burst_ratio` 是否能降低无 request 或 request 偏低导致的节点压力低估。

第五，友商横向对比场景。在 CCE 和 ACK 构建相同任务模型，对比节点分布、热点持续时间、OOM 风险和业务恢复时间。

## 演进方向

这次增强主要围绕 CPU 和内存两个维度，解决监控延迟窗口内的负载估算与节点打分问题。后续还有几个演进方向。

第一，从 CPU/Mem 扩展到智算硬件真实利用率。AI 训练、推理、视频处理等场景中，GPU/NPU 利用率、显存使用率、设备温度、硬件队列等待等指标同样会影响调度质量。后续可以把影子负载模型扩展到智算硬件维度，让调度器更准确地感知异构资源压力。

第二，引入任务画像和资源使用预测。当前估算主要基于 request、limit、QoS 和配置参数。未来可以结合历史运行数据、任务类型、队列信息，为不同业务生成更精细的资源画像，从静态估算走向预测式估算。

第三，引入节点池粒度的负载感知调度。实际生产集群往往按节点池区分业务、规格、可用区或成本模型。后续可以在节点池层面引入负载视图，避免单个节点维度优化导致节点池整体冷热不均。

第四，与重调度能力联动。调度阶段可以避免制造新热点，重调度阶段可以治理已经形成的热点。两者结合，才能形成完整的负载治理闭环。

第五，增强可观测性。Shadow Load 本身也可以暴露为调度器内部指标，让用户看到某个节点为什么降分、哪些 Pod 正在贡献影子负载、风险系数何时生效。这对运维排障和客户信任都很重要。

## 总结

这次 Volcano 负载感知调度增强，来自真实客户迁移和日常发布过程中的高频问题：批量新建和重部署 Pod 时，调度器容易持续选择当前打分最优节点，导致新热点形成。

问题的根因不是调度器完全没有负载感知，而是外部指标存在监控延迟。新 Pod 已经被调度器分配出去，但其资源消耗尚未进入节点利用率指标。调度器如果只看外部指标，就会在短时间内低估目标节点压力。

增强后的方案通过资源预估和影子负载，将“调度器已经知道的新 Pod”提前计入节点打分；通过快照机制，在 Gang 调度和回滚场景中保证加减一致；通过可配置参数，在不同客户业务中平衡保守性和资源利用率。

从结果看，这个特性让可观测数据真正进入调度闭环：外部指标提供真实负载，调度事件提供尚未上报的新负载，调度器把两者合并后做出更接近当前状态的节点选择。
