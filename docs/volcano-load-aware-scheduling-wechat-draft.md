# 让可观测数据进入调度闭环：Volcano 负载感知调度增强实践

在负载感知调度里，“当前最空闲的节点”并不一定适合继续承接下一批 Pod。

原因不复杂。节点 CPU、内存利用率来自监控系统，而监控系统有采集、聚合、上报和拉取周期。调度器做决策时，看到的是上一段时间的结果；但批量新建、滚动重部署这类场景里，调度动作会连续发生。一个 Pod 刚被调度到某个节点，调度器已经知道这件事，节点指标却还没来得及变化。如果后续 Pod 仍然只看外部指标，就容易反复落到同一个“看起来还很空”的节点上。

这次 Volcano 负载感知调度增强，处理的正是这个时间差。它没有推翻原有 usage 插件的模型，而是在调度器内部补了一层短时记忆：已经调度出去、但还没有体现在监控曲线里的 Pod，会先按一定规则估算资源占用，并临时计入节点打分。这样，调度器下一次选择节点时，不会再把这些 Pod 当作不存在。

## 真实客户需求

这个问题来自真实生产现场。

xx客户在将主站业务从友商迁移到华为云时，在 CCE 集群中使用 Volcano 负载感知调度插件。迁移和后续发布过程中，运维侧发现一个很具体的现象：批量新建 Pod 或重部署 Pod 时，Volcano 会把大量 Pod 调度到当时打分最高的节点上。短时间内，这个节点的 CPU、内存指标仍然偏低，调度分数也还不错；等新 Pod 启动并开始消耗资源后，这个节点很快就可能变成新的热点。

客户反馈，友商产品的调度插件在节点打分时会考虑一种情况：节点上已经有新 Pod 被调度过去，但这些 Pod 的利用率还没有上报到监控系统。调度器会据此调整节点分数，避免后续 Pod 继续集中落到同一个节点。相比之下，CCE 当前 Volcano 插件缺少这段判断，导致客户在主站迁移、日常重部署和扩缩容时反复遇到类似问题。

这个需求的紧急程度高，不是偶发边缘问题。客户每天都会进行发布或重部署，直接受影响的是负责迁移、发布、扩缩容和集群稳定性的运维人员。典型场景包括批量新建 Pod、滚动重部署、节点扩容后的任务回填，以及业务高峰前的集中发布。

所以，这里要解决的不是“让 Pod 数量看起来更平均”。真正的问题是：负载感知调度已经能看到节点真实利用率，但它还没有把“刚刚调度出去的新负载”算进来。在监控指标刷新之前，调度器会短暂低估目标节点的压力，而这个短暂窗口在批量调度时足以制造热点。

## 背景导入：上一篇解决了什么，这一篇继续解决什么

上一篇负载感知调度文章主要讲 usage 插件如何让 Volcano 在调度时看到节点真实负载。过去调度器更多依赖 Pod request、limit 和节点 allocatable；接入 Prometheus、Custom Metrics API、Elasticsearch 等指标源后，usage 插件可以把运行时 CPU、内存利用率纳入调度决策。

这个能力主要落在两个阶段。

Predicate 阶段会用用户配置的 CPU、内存阈值过滤节点。如果某个节点真实利用率已经超过阈值，就不再继续向它调度新 Pod。

NodeOrder 阶段会根据节点负载打分。负载越低，得分越高；节点越忙，得分越低。这样，调度器不再只看资源声明，而是可以优先选择运行时更空闲的节点。

这解决了“如何基于真实负载调度”的问题。但客户场景暴露出另一个细节：调度器看到的真实负载，仍然不是调度瞬间的完整负载。外部指标有采集周期，调度器也有拉取周期。一个 Pod 被绑定到节点之后，调度器马上知道这个事件发生了，Prometheus 或 Custom Metrics API 却要等下一轮采集后才能反映出来。

单个 Pod 调度时，这个时间差通常不明显；批量新建、滚动重部署、Gang 调度时，它会被放大。多个 Pod 在一个或几个调度周期里连续完成节点选择，如果某个节点在外部指标里仍然“很空”，它就可能被连续选中。等指标上报时，热点已经形成。

这一篇讨论的增强，就是在 usage-based scheduling 的基础上，把“已经调度但尚未被监控系统看见的新 Pod”也纳入节点负载判断。上一篇让调度器看见外部可观测数据，这一篇让调度器把自己的最新调度动作也记进这份负载视图。

## 技术选型：为什么选择节点视角的影子负载方案

设计这个能力时，最直接的想法是从 Pod 视角做判断：根据每个 Pod 的业务类型、历史画像、实际资源使用情况，推断它更适合哪个节点。这个方向足够精细，但并不适合先作为通用能力落地。新 Pod 刚启动时没有实际利用率，历史画像也不总是可得；不同客户的业务波动方式差异很大，把调度逻辑绑定到业务语义上，会让一个通用调度插件变成难以维护的预测系统。

也可以对节点做固定惩罚：只要当前周期内给某个节点分配过新 Pod，就扣一定分。这种实现简单，但很快会遇到精度问题。一个 50m CPU 的小 Pod 和一个 4 核 CPU 的大 Pod，不应该带来相同惩罚；Guaranteed、Burstable、BestEffort 之间的资源声明也不同。固定扣分不是太保守，就是难以拦住真正的大负载。

还有一种方向是做更强的分散，把 Pod 尽量摊到不同节点上。它对某些场景有帮助，但它不是负载感知。节点容量不同，当前 CPU/Mem 压力不同，Pod 数量平均不代表负载平均。客户遇到的问题也不是“Pod 个数不均”，而是“新增负载没有及时反映到节点分数里”。

所以这次没有转向 Pod 画像或固定惩罚，而是沿着 usage 插件原来的节点视角往前补。调度器仍然以节点为打分对象，仍然使用 CPU/Mem 真实利用率，也仍然在 Predicate 和 NodeOrder 中完成决策。新增的部分是一份内部维护的影子负载：某个 Pod 已经被调度到节点、但资源消耗还没进入外部指标时，调度器先为它估算一份 CPU/Mem 占用，临时记到这个节点上。

这个选择有几个实际好处。它和 usage 插件原来的模型一致，不需要引入新的业务语义；对客户业务没有侵入，不要求改 Pod 模板或维护任务画像；同时它比固定扣分更容易解释，因为每个节点降分都可以拆回真实负载、影子负载和风险系数。

## 怎么做：原理介绍

这次增强主要由三块组成：资源预估、影子负载，以及调度事件触发的新增和回退。

### 资源预估

对 Guaranteed 和 Burstable Pod，插件会根据 request、limit 和配置权重估算资源消耗：

```text
pod_estimate = (request * request_ratio + (limit - request) * burst_ratio) * applied_risk_factor
```

`request_ratio` 表示 request 部分在估算值里的权重，`burst_ratio` 表示 request 到 limit 之间弹性空间的权重。默认配置更相信 request，burst 部分默认不计入；如果业务启动期或高峰期突发明显，可以适当提高 `burst_ratio`。

估算值会被限制在合理范围内，不会低于 0，也不会超过有效 limit。如果某个资源没有 limit，或者 limit 小于 request，就把 request 当作有效上限。这样做是为了避免无 limit Pod 被夸大估算，也避免异常资源声明把节点分数带偏。

BestEffort Pod 没有 request 和 limit，不能按上面的公式计算。插件为它提供固定估算值，默认 CPU 为 `250m`，内存为 `200Mi`。这两个值可以结合客户集群里的历史运行情况调整，至少避免 BestEffort Pod 在节点打分时被完全忽略。

### 风险系数

节点越接近高水位，同样一个新 Pod 带来的风险越大。因此插件引入了 `risk_threshold` 和 `risk_factor`。

调度器会先把真实负载和影子负载合起来，计算 CPU、内存维度的复合利用率：

```text
cpu_composite = (real_cpu_load + shadow_cpu_estimate) / node_cpu_capacity
memory_composite = (real_memory_load + shadow_memory_estimate) / node_memory_capacity
load_composite = weighted_average(cpu_composite, memory_composite)
```

当 `load_composite` 达到 `risk_threshold` 后，后续新增 Pod 的估算值会乘以 `risk_factor`。这不是为了把节点一刀切过滤掉，而是让节点接近风险水位时，调度器在排序阶段更谨慎。

### 影子负载

影子负载是 session 级别的节点估算负载缓存。每个节点维护 CPU 和内存两类累计值。每个被计入缓存的 Pod，也会保存一份快照，记录它当时估算出来的 CPU、内存，以及落到哪个节点。

NodeOrder 打分时，节点使用的不再只是外部指标里的：

```text
real_load
```

而是：

```text
real_load + shadow_load
```

这样，在外部指标还没上报之前，调度器也知道某个节点刚刚已经承接了一批新 Pod，后续打分会自然下降。

### 新增和回退

Pod 在当前调度 session 中被 Allocate 到某个节点时，usage 插件会估算它的 CPU/Mem 消耗，并把估算值加入该节点的影子负载。

如果后续发生 Deallocate，也就是这次临时调度需要回滚，插件不会重新计算一次估算值，而是直接使用 Allocate 时保存的快照做扣减。这个细节对 Gang 调度很重要。一组 Pod 如果不能整体调度成功，前面临时分配出去的 Pod 需要全部回滚；影子负载如果不跟着精确回退，后续调度就会高估节点压力。

所以实现上采用的是“加的时候保存快照，减的时候按快照扣回”。即使加减之间节点风险水位变化、配置变化或估算逻辑重新触发，也不会影响回滚一致性。

## 怎么实操：工程实现中的细节

### 如何判断节点是否过热

节点过热判断分成两层。

Predicate 阶段仍然只看真实监控指标。如果节点真实 CPU 或内存利用率超过用户配置的 `thresholds.cpu` 或 `thresholds.mem`，节点会被过滤。这条规则保持简单，避免因为估算值偏保守而误杀节点。

NodeOrder 阶段会把真实负载和影子负载合并成复合利用率，再计算节点得分。复合利用率越高，节点得分越低。复合利用率超过 `risk_threshold` 后，新增 Pod 的估算还会乘以 `risk_factor`，让后续调度更谨慎。

这样拆开以后，“不能调度”的判断仍然交给真实指标，“不应该继续优先调度”的判断由组合负载影响。对生产集群来说，这比直接把估算值用于硬过滤更稳。

### 为什么 Pod 预估值这样设计

Pod 预估值没有简单取 request，也没有直接取 limit，而是通过 `request_ratio` 和 `burst_ratio` 组合出来。

现实里，不同客户的资源声明质量差异很大。有些业务 request 配得比较准，稳定运行时和 request 接近；有些业务 request 偏低，启动期或高峰期会明显冲高。如果完全按 request 估算，容易低估突发业务；如果完全按 limit 估算，又可能把节点分数压得过低，降低集群利用率。两个比例参数给了用户调整空间。

默认值偏稳妥：request 是主要依据，burst 部分默认不计入，避免所有 Pod 都按 limit 估算后过度分散。BestEffort Pod 则通过 `be_cpu` 和 `be_mem` 配置固定估算值，解决没有 request/limit 时完全无法入账的问题。

实现里还处理了几个容易出错的边界：

- request 为负数时按 0 处理；
- limit 缺失或小于 request 时，以 request 作为有效 limit；
- 估算结果被限制在 `[0, effectiveLimit]` 范围内；
- initContainers 按 Kubernetes 资源语义取最大值，普通 containers 按总和计算；
- `risk_factor` 小于 1 时不会降低估算值，避免高压场景下反而更激进。

这些处理让参数可调，但不至于因为个别异常配置得出明显不合理的节点分数。

### 如何处理 Gang 调度和回滚

Volcano 经常用于批处理和 AI 任务，Gang 调度是其中很典型的能力：一组 Pod 要么整体调度成功，要么整体等待。调度器可能先给其中一部分 Pod 选好了节点，最后发现整个 PodGroup 条件不满足，于是需要回滚。

如果影子负载只在 Allocate 时增加，不在 Deallocate 时准确扣回，就会留下两类问题。已经回滚的 Pod 并不会真的运行，但节点缓存里还留着它的估算负载；后续调度会被这份不存在的负载影响，错误地避开某些节点。

为避免这种情况，ShadowLoadCache 为每个 Pod 保存 `PodEstSnapshot`。快照里有节点名、CPU 估算值和内存估算值。Deallocate 时直接按快照扣减，不重新估算。这样即使节点压力、风险系数或配置在加减之间发生变化，也不会破坏加减一致性。

### 如何保证影子负载缓存不低估也不高估

影子负载最怕两种错误：低估会让热点继续出现，高估会让调度器长期避开本来可用的节点。

实现上先在 session 打开时初始化 ShadowLoadCache，并读取 `metrics.interval` 作为监控延迟窗口。插件会扫描当前节点上的任务，把已经分配到节点、但可能还没被指标系统覆盖的 Pod 加入缓存。这里包括 Allocated、Binding、Bound，以及启动时间短于 metrics interval 的 Running Pod。

Pending、Succeeded、Failed 不会进入影子负载。还没绑定节点、已经成功退出或失败退出的任务，都不应该再影响节点估算。

同一个 task 如果重复加入缓存，旧快照会先被扣掉，再写入新快照，避免重复计算。ShadowLoadCache 读写使用锁保护，防止并发访问下出现负值或状态不一致。session 关闭时缓存会被清理，避免跨 session 残留。

### 为什么影子负载不用于硬过滤

这次实现里，Predicate 仍然只检查真实指标是否超过阈值，影子负载主要影响 NodeOrder 排序。这是有意保留的边界。

影子负载毕竟是估算值。估算适合影响优先级，但如果直接用于过滤，配置偏保守时可能过早排除节点，影响任务可调度性。尤其是 Gang 调度，过度过滤会放大整组任务调度失败的概率。

所以我们把“已经明确过载”的判断交给真实指标，把“这台节点接下来不该再优先选”的判断交给组合负载。这样既保留了负载感知调度的安全边界，也让增强能力以更平滑的方式进入节点排序。

## 配置示例

这次增强延续 usage 插件已有配置方式，同时新增 estimator 相关参数：

```yaml
actions: "enqueue, allocate, backfill"
tiers:
  - plugins:
      - name: usage
        enablePredicate: false
        arguments:
          usage.weight: 5
          cpu.weight: 1
          memory.weight: 1
          thresholds:
            cpu: 80
            mem: 80
          estimator:
            request_ratio: 0.7
            burst_ratio: 0
            risk_threshold: 0.6
            risk_factor: 1.2
            be_cpu: 250m
            be_mem: 200Mi
metrics:
  type: prometheus
  address: http://prometheus:9090
  interval: 30s
```

这里的 `metrics.interval` 不只影响指标拉取频率，也会用于判断监控盲区。对刚启动不久的 Running Pod，如果运行时间还没有覆盖这个 interval，插件会认为它仍可能没有进入外部指标，并把它纳入影子负载。

调参时可以从业务特征出发。稳态业务可以保持较高的 `request_ratio`，让估算接近资源声明；突发业务可以适当提高 `burst_ratio`，降低 request 偏低带来的低估；对热点风险更敏感的业务，可以降低 `risk_threshold` 或提高 `risk_factor`；BestEffort 较多的集群，则需要结合历史运行数据校准 `be_cpu` 和 `be_mem`。

## 痛点、挑战、方案、亮点

把前面的内容压缩来看，这个增强要处理的是一个很具体的调度盲区。

痛点在监控延迟。客户已经有负载感知调度，但在批量调度时，真实指标到达得比调度决策慢。当前最优节点连续接收新 Pod 后，可能很快从低负载变成热点。

挑战在一致性和尺度感。调度器需要提前考虑新 Pod 的资源影响，但不能把估算值用得过重；Gang 调度回滚时，影子负载必须精确扣回；参数要允许客户按业务差异调优，又不能因为配置异常让分数失真。

方案是节点视角的影子负载。它沿用 usage 插件的节点打分模型，用资源预估计算新 Pod 对 CPU/Mem 的影响，再把估算值写入 session 级 ShadowLoadCache。后续 NodeOrder 使用真实负载和影子负载共同计算分数，Deallocate 时按快照回退。

这版实现最值得保留的是三点：业务无侵入，节点降分可解释，调度回滚不留下残留负载。它不要求客户维护复杂任务画像，也不把估算值变成硬过滤条件，而是把调度器已经知道、监控系统暂时还不知道的信息补到排序阶段。

## 方案选型与友商对比

客户反馈中提到，阿里云侧调度插件在节点打分时会考虑“利用率尚未上报的新 Pod”，避免过量调度产生热点。这说明客户关注的不是简单打散 Pod，而是调度器能不能理解监控延迟窗口里的新增负载。

在增强后，我们基于客户真实生产环境信息搭建模拟环境，并在友商集群上构建相同任务模型做横向验证。现有验证结论可以这样表达：大多数场景下，增强后的 Volcano 负载感知调度与友商方案表现持平，部分场景略占优；在少数场景里，友商方案当前实现没有覆盖到对应风险，可能出现 OOM，而增强后的 Volcano 可以通过影子负载和准确的资源预估提前降低热点节点得分，减少继续集中调度的概率。

从实现路径看，这次增强是在 Volcano usage 插件已有结构里补齐一段缺失信息。真实指标仍来自 metrics source，硬阈值仍在 Predicate 生效，节点排序仍在 NodeOrder 完成；影子负载只负责记录“调度器已知但监控尚未知”的新负载。

这个路径对 Volcano 的批处理和 Gang 调度更友好。它不只处理单个 Pod 的连续调度，还处理 PodGroup 整体调度失败后的回滚一致性。快照式扣减就是为这类场景保留的工程细节。

## 落地效果如何验证

这类调度增强不能只看某一次 Pod 落到了哪里，更要看连续调度过程中节点负载和分数怎么变化。

测试报告可以围绕几类场景展开。

- 批量新建：连续创建一批 Pod，观察增强前是否仍集中落到最初打分最高的节点，增强后节点分数是否会随影子负载下降。
- 重部署：模拟客户日常发布，观察滚动重建过程中是否出现局部节点负载快速升高，增强后热点是否被削弱。
- Gang 调度：构造部分 Pod 临时分配后整体调度失败的 PodGroup，验证 Deallocate 后影子负载是否准确回退。
- BestEffort 或 request 偏低：验证 `be_cpu`、`be_mem`、`burst_ratio` 是否能降低无 request 或 request 偏低导致的节点压力低估。
- 友商横向对比：在 CCE 和 ACK 构建相同任务模型，对比节点分布、热点持续时间、OOM 风险和业务恢复时间。

## 演进方向

这次增强先围绕 CPU 和内存，解决监控延迟窗口内的负载估算和节点打分问题。后续可以继续往几个方向推进。

智算硬件真实利用率需要进入同一套逻辑。AI 训练、推理、视频处理等场景里，GPU/NPU 利用率、显存使用率、设备温度、硬件队列等待时间都会影响调度质量。未来可以把影子负载模型扩展到异构资源维度，让调度器更早感知 GPU/NPU 压力。

任务画像和资源使用预测也值得引入。当前估算主要依赖 request、limit、QoS 和配置参数，属于相对通用的静态估算。后续如果能结合历史运行数据、任务类型、队列信息，就可以为不同业务生成更贴近实际的资源画像。

节点池粒度的负载感知也有价值。生产集群往往按节点池区分业务、规格、可用区或成本模型。只在单节点维度做优化，可能仍然留下节点池层面的冷热不均。后续可以在节点池层面维护负载视图，再把节点池选择和节点选择结合起来。

调度和重调度可以进一步联动。调度阶段避免制造新热点，重调度阶段治理已经形成的热点。两者配合起来，才更接近完整的负载治理链路。

影子负载自身也应该更可观测。后续可以把 Shadow Load 暴露为调度器内部指标，让用户看到节点为什么降分、哪些 Pod 正在贡献影子负载、风险系数什么时候生效。这对运维排障和客户信任都很重要。

## 总结

这次 Volcano 负载感知调度增强，源头是客户在迁移和日常发布中反复遇到的热点问题：批量新建或重部署 Pod 时，调度器容易持续选择当前打分最高的节点，最终把新负载堆到同一处。

根因不是调度器完全没有负载感知，而是外部指标存在延迟。新 Pod 已经被调度器分配出去，但资源消耗尚未进入节点利用率曲线。只看外部指标，调度器就会在短时间内低估目标节点压力。

增强后的方案用资源预估和影子负载补上这段时间差。调度器把自己刚刚分配出去的新 Pod 临时记入节点打分；发生 Gang 调度回滚时，再按快照精确扣回；不同业务则通过 `request_ratio`、`burst_ratio`、`risk_threshold`、`risk_factor`、`be_cpu`、`be_mem` 调整估算策略。

对用户来说，这个能力的价值很直接：在监控数据还没刷新前，调度器也能更早意识到某些节点已经承接了新负载，从而减少批量发布和重部署时制造新热点的概率。
