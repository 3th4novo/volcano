# 让可观测数据进入调度闭环：Volcano 负载感知调度增强实践

在 Kubernetes 集群里，调度器看到的“资源空闲”，有时和业务真正感受到的“节点不忙”并不是一回事。

很多集群都遇到过类似现象：某些节点的 request 分配率并不高，但业务一跑起来 CPU 或内存已经很紧张；另一些节点 request 看起来被占满，真实负载却长期偏低。到了批量发布、滚动重部署、节点扩容回填任务时，这个偏差会被进一步放大。调度器连续选择当前看起来最优的节点，新 Pod 启动后又把这些节点推成热点，运维侧看到的是发布后局部节点 CPU/Mem 飙高、业务抖动，严重时甚至出现 OOM。

这就是负载感知调度要解决的问题。它不是简单追求 Pod 数量平均，而是希望调度器在放置 Pod 时更接近节点的真实运行压力。

## 从客户现场看：负载感知调度到底解决什么问题

真实生产集群里的业务资源使用并不规整。CPU 密集型服务、内存敏感服务、启动期突发型任务、BestEffort 任务可能运行在同一批节点上。即使团队已经要求业务填写 request 和 limit，资源声明也很难长期准确反映运行时消耗。

这会带来几类常见问题。

有些业务 request 配得偏低，调度器认为节点仍有不少可分配资源，于是继续把 Pod 放上去；但业务运行后真实 CPU/Mem 消耗很高，节点很快过热。有些业务 request 配得偏高，调度器认为节点已经很满，实际资源却还没被充分使用，集群利用率上不去。还有一些业务本身波动大，平时消耗低，高峰或启动阶段突然冲高，只看静态资源声明很难提前识别风险。

之前公众号文章《基于 Volcano 实现节点真实负载感知调度》已经讲过这个问题：默认调度器主要根据 Pod request 和节点 allocatable 计算可调度资源。这个模型对 Kubernetes 的资源语义很重要，但它并不会天然知道节点此刻真实 CPU、内存利用率。结果就是，调度器可能把新 Pod 放到 request 分配率较低、但真实负载已经很高的节点上。

这类问题在单个 Pod 调度时不一定明显，但在批量新建、滚动重部署、扩缩容和业务迁移时会变得很突出。客户在将主站业务从友商迁移到华为云 CCE 集群时，就遇到了类似问题：批量新建或重部署 Pod 时，调度器会大量选择当时打分最高的节点。短时间内，这个节点的监控指标还没有明显变化；等新 Pod 启动并开始消耗资源后，它已经成为新的热点。

这里还有一个更细的时间差。即使调度器已经接入 Prometheus、Custom Metrics API 或云原生监控插件，真实指标也有采集、聚合、上报和拉取周期。调度器刚把 Pod 分配到某个节点时，它自己已经知道这个调度事件发生了，但监控系统还没来得及把新 Pod 的消耗反映到节点曲线上。如果后续 Pod 仍然只看外部指标，就可能在这个窗口里继续选择同一个节点。

所以，负载感知调度要解决的是两层问题：一是让调度器从“只看资源声明”走向“参考真实负载”；二是在监控指标滞后的窗口里，让调度器也能把自己刚刚做出的调度决策纳入判断。

## 业界一般怎么解决这类问题

目前业界解决这类问题，大致可以分成队列准入和节点负载感知两类。

Kueue 更偏向队列准入和配额管理。它通过 ClusterQueue、ResourceFlavor、nominalQuota、borrowingLimit、lendingLimit 等机制，控制 Workload 是否可以进入集群，以及不同队列之间如何共享资源。这个方向适合批任务准入、公平共享和资源配额治理，解决的是“作业能不能进来、队列之间怎么分资源”的问题。

Koordinator 更偏向节点视角的负载感知调度。它通过 NodeMetric 获取节点真实负载，在调度时过滤高负载节点、优先选择低负载节点，并对指标延迟窗口内的新 Pod 做资源估算。它代表了业界比较成熟的一类方案：在节点维度引入真实运行指标，同时通过估算弥补新 Pod 尚未进入指标的时间差。

各云厂商的 ACK、CCE 等托管 Kubernetes 产品，也通常会围绕真实指标、节点打分、阈值过滤、Pod 资源估算等方向增强调度能力。背后的共识是明确的：生产集群调度不能只看 request；队列层要管准入和配额，节点层要看运行时负载，而批量调度场景还要处理指标延迟。

Volcano 这次负载感知调度增强，正是沿着节点运行时负载这条线，把真实指标、资源预估、影子负载和调度回滚一致性放到同一套调度路径里。

## Volcano 的完整方案：真实负载、影子负载与资源预估

Volcano usage 插件的目标，是让调度器在选择节点时参考节点真实负载，并在监控延迟窗口内补上调度器已经知道、监控系统暂时还不知道的新负载。

这套方案由几部分组成。

### 指标来源：让调度器看见真实负载

usage 插件可以接入 Prometheus、Custom Metrics API、云原生监控插件等指标源，把节点真实 CPU/Mem 利用率带入调度流程。

这样，调度器不再只根据 Pod request、limit 和节点 allocatable 判断节点是否合适，而是可以把运行时负载也纳入判断。对用户来说，最直接的变化是：如果某个节点 request 分配率不高但真实 CPU 已经很高，调度器可以识别到这个风险，避免继续把新 Pod 放上去。

### 节点过滤：真实过热节点不再继续承接新 Pod

Predicate 阶段会根据用户配置的 `thresholds.cpu` 和 `thresholds.mem` 判断节点真实负载是否超过水位。

如果节点 CPU 或内存真实利用率已经超过阈值，插件可以阻止新 Pod 继续调度到该节点。这里使用真实指标作为硬边界，原因很直接：硬过滤应该尽量基于已经观测到的事实，避免因为估算偏保守而过早排除可用节点。

### 节点打分：负载越低，优先级越高

NodeOrder 阶段会基于节点 CPU/Mem 负载计算得分。节点负载越低，得分越高；节点压力越高，得分越低。

CPU 和 Memory 权重可以配置。如果业务更容易被 CPU 打满，可以提高 CPU 权重；如果业务对内存更敏感，可以提高 Memory 权重。这样调度器不是机械地看一个统一分数，而是可以结合业务资源特征调整排序结果。

### 影子负载：把“监控还没看见的新 Pod”先记进来

真实指标有延迟。一个 Pod 刚刚被调度到某个节点后，调度器已经知道这件事，但外部监控可能还要等下一轮采样和聚合后才会反映出来。

为了解决这个窗口，usage 插件维护了一份 session 级别的 Shadow Load。已经调度到节点、但还没有进入监控指标的新 Pod，会先被估算出一份 CPU/Mem 占用，并临时记到对应节点上。后续 NodeOrder 打分时，节点使用的是：

```text
真实负载 + 影子负载
```

而不是只看外部指标里的真实负载。

`metrics.interval` 会作为监控盲区判断的参考。插件会把 Allocated、Binding、Bound 状态的任务，以及启动时间短于 `metrics.interval` 的 Running Pod 纳入影子负载考虑。Pending、Succeeded、Failed 等不应该影响节点压力的任务，则不会被计入。

这对批量发布和滚动重部署尤其关键。调度器不会再连续把 Pod 放到一个“指标还没变化”的节点上，而是会随着影子负载增加，逐步降低该节点分数。

### 资源预估：让不同类型 Pod 都能参与节点评分

影子负载要发挥作用，首先要知道新 Pod 应该估算多少资源。usage 插件对 Guaranteed 和 Burstable Pod 使用 request、limit 和可配置权重估算资源消耗：

```text
pod_estimate = (request * request_ratio + (limit - request) * burst_ratio) * applied_risk_factor
```

`request_ratio` 表示 request 部分在估算值里的权重，`burst_ratio` 表示 request 到 limit 之间弹性空间的权重。默认配置更相信 request，避免所有 Pod 都按 limit 估算后过度分散；如果业务启动期或高峰期突发明显，可以提高 `burst_ratio`，让调度器更重视弹性空间带来的风险。

估算值会被限制在合理范围内，不会低于 0，也不会超过有效 limit。如果某个资源没有 limit，或者 limit 小于 request，则以 request 作为有效上限。这样可以避免异常资源声明把节点分数带偏。

BestEffort Pod 没有 request 和 limit。为了避免这类 Pod 在节点打分时完全“免费”，插件提供 `be_cpu` 和 `be_mem` 作为默认估算值。默认值分别是 `250m` 和 `200Mi`，用户可以根据集群中 BestEffort 任务的真实表现调整。

### 风险系数：节点接近高水位时更保守

节点越接近高水位，同样一个新 Pod 带来的风险越大。usage 插件引入 `risk_threshold` 和 `risk_factor` 来表达这一点。

插件会先计算 CPU/Mem 维度的复合负载。当复合负载达到 `risk_threshold` 后，后续新增 Pod 的估算值会乘以 `risk_factor`。这不会直接把节点过滤掉，而是让节点在排序阶段更谨慎地降分。

换句话说，真实指标负责判断节点是否已经明确过热，风险系数负责在节点接近高水位时提前降低继续调度的倾向。

### 调度事件一致性：Gang 回滚不能留下残留负载

Volcano 面向批处理、AI 训练和推理等场景，Gang 调度是重要能力。一组 Pod 可能需要整体调度成功才有意义；如果部分 Pod 已经临时分配了节点，但最终 PodGroup 无法整体满足条件，就需要回滚。

影子负载如果只在 Allocate 时增加，不在 Deallocate 时精确扣回，就会留下残留估算负载。后续调度会误以为某些节点已经承接了不存在的负载，从而错误避开这些节点。

因此，Shadow Load 在加入估算值时会保存快照，记录 Pod 当时估算出的 CPU、内存和目标节点。发生 Deallocate 时，插件按快照做精确扣减，而不是重新计算一遍估算值。这样即使节点压力、风险系数或配置在加减之间发生变化，也不会破坏回滚一致性。

这也是 Volcano 方案在批处理和 Gang 调度场景里的关键竞争力：它不只关心单个 Pod 的放置结果，也关心调度过程中的临时分配和失败回滚是否会污染下一轮决策。

### 方案竞争力：在调度路径里提前规避热点

综合来看，Volcano 负载感知调度不是只做队列准入，也不是等热点形成后再治理，而是在调度路径中提前规避热点。

它对业务无侵入，不要求用户维护复杂任务画像；它基于节点真实负载和调度器内部事件做判断，节点为什么降分可以从真实负载、影子负载、资源权重和风险系数中解释；它支持用户按业务特征调参，也能适配 Volcano 批处理、AI、Gang 调度等典型场景。

对客户来说，这套能力最终带来的体验是：批量发布和重部署时，调度结果更平滑，局部节点过热概率更低；当业务资源声明不够准确时，也可以通过配置把经验沉淀到调度策略里。

## 用户如何结合业务深度调优

负载感知调度的一个重要价值，是允许用户结合业务特征做调优，而不是所有集群都使用一套固定策略。

下面是当前文章建议呈现的参数说明。默认值以当前实现和示例配置为准。

```yaml
actions: "enqueue, allocate, backfill"
tiers:
  - plugins:
      - name: usage
        enablePredicate: false
        arguments:
          usage.weight: 5
          cpu.weight: 1
          memory.weight: 1
          thresholds:
            cpu: 80
            mem: 80
          estimator:
            request_ratio: 0.7
            burst_ratio: 0
            risk_threshold: 0.6
            risk_factor: 1.2
            be_cpu: 250m
            be_mem: 200Mi
metrics:
  type: prometheus
  address: http://prometheus:9090
  interval: 30s
```

`thresholds.cpu` 和 `thresholds.mem` 默认建议为 `80` / `80`，用于控制节点真实负载水位。稳定性优先的业务可以适当降低阈值，让节点更早退出候选；资源利用率优先的场景，可以在评估风险后适当提高。

`cpu.weight` 和 `memory.weight` 默认都是 `1`。CPU 密集型业务可以提高 CPU 权重，内存敏感型业务可以提高 Memory 权重。比如视频处理、编译、推理前处理等任务可能更关注 CPU；缓存、数据处理、大模型服务则可能更关注内存。

`request_ratio` 默认是 `0.7`。如果业务 request 配置较准，可以保持默认或适当提高，让估算更接近稳定运行时的资源声明。

`burst_ratio` 默认是 `0`。如果业务启动期突发明显，或者长期存在 request 偏低、limit 更能代表风险上限的情况，可以适当提高 `burst_ratio`，让调度器更保守地看待弹性空间。

`risk_threshold` 默认是 `0.6`。如果希望节点接近中高水位时更早降分，可以降低这个值；如果更追求利用率，可以适当提高，但需要结合业务稳定性验证。

`risk_factor` 默认是 `1.2`。业务对热点敏感、OOM 风险高、发布批次大的场景，可以提高这个值，让节点进入风险水位后更明显降分。

`be_cpu` 默认是 `250m`，`be_mem` 默认是 `200Mi`。如果集群里 BestEffort 任务较多，且这些任务实际消耗并不低，就需要结合历史监控校准这两个值，避免 BestEffort Pod 在调度时被完全低估。

`metrics.interval` 建议与监控采集/聚合周期保持一致，例如 `30s`。监控周期较长时可以增大，避免刚启动的 Pod 过早从影子负载中移出；监控更实时的场景可以适当减小，让影子负载更快交还给真实指标。

调参时可以先按业务类型分层：稳态业务优先校准 `request_ratio`；突发业务重点观察 `burst_ratio` 和 `risk_factor`；BestEffort 较多的集群重点校准 `be_cpu`、`be_mem`；发布和重部署风险较高的业务重点关注 `risk_threshold`、`metrics.interval` 和节点真实阈值。

## 方案收益与测试数据

这类调度增强不能只看某一次 Pod 落到了哪里，更要看连续调度过程中节点负载和分数怎么变化。

建议在正式发布前，把测试报告按以下场景补充进来。

### 批量新建 Pod

重点观察增强前后 Pod 是否仍集中落到少数节点，最高负载节点 CPU/Mem 峰值是否下降，节点负载曲线是否更平滑。

[待补充：Pod 数量、节点规格、批量创建速率、增强前后节点分布、最高 CPU/Mem 峰值、热点持续时间。]

### 滚动重部署

模拟客户日常发布过程，观察重部署期间局部节点是否快速升温，以及增强后热点是否被削弱。

[待补充：Deployment 副本数、滚动批次、maxSurge/maxUnavailable、增强前后节点负载曲线、业务恢复时间。]

### request 偏低场景

构造 request 明显低于真实消耗的 Pod，验证 `burst_ratio` 是否能降低资源低估风险。

[待补充：request/limit 配置、实际资源消耗、不同 `burst_ratio` 下节点分布和负载峰值。]

### BestEffort 场景

构造无 request/limit 的 BestEffort Pod，验证 `be_cpu` 和 `be_mem` 是否能让它们进入调度评分，减少热点和 OOM 风险。

[待补充：BestEffort Pod 数量、默认估算值与调优值对比、OOM 复现条件、增强后结果。]

### Gang 调度回滚

构造部分 Pod 临时分配后整体调度失败的 PodGroup，验证 Deallocate 后影子负载是否准确回退，不影响下一轮调度。

[待补充：PodGroup 规模、minAvailable、失败触发条件、回滚前后 Shadow Load 数值、下一轮调度结果。]

### 友商横向对比

在 CCE 和友商 ACK 集群中构建相同任务模型，对比节点分布、负载峰值、热点持续时间、OOM 风险和业务恢复时间。

[待补充：相同节点规格、相同 Pod 模型、相同发布批次下的对比数据。结论建议围绕“大多数场景持平或略优，小部分场景 Volcano 增强后覆盖更完整”展开。]

## 未来演进方向

这次增强先围绕 CPU 和内存，解决监控延迟窗口内的负载估算和节点打分问题。后续还可以继续向几个方向演进。

首先是扩展到智算硬件真实利用率。AI 训练、推理、视频处理等场景中，GPU/NPU 利用率、显存使用率、设备队列等待、设备温度等指标都会影响调度质量。未来可以把影子负载和风险估算扩展到异构资源维度，让调度器更早感知 GPU/NPU 压力。

其次是引入任务画像和资源使用预测。当前估算主要基于 request、limit、QoS 和配置参数，属于通用静态估算。后续可以结合历史运行数据、任务类型、队列信息，为不同业务生成更贴近实际的资源画像，让估算从“可配置经验值”走向“基于历史行为的预测”。

节点池粒度的负载感知也值得推进。生产集群往往按节点池区分业务、规格、可用区或成本模型。只在单节点维度优化，可能仍然留下节点池层面的冷热不均。后续可以在节点池层面维护负载视图，再把节点池选择和节点选择结合起来。

调度和重调度也可以进一步联动。调度阶段负责避免制造新热点，重调度阶段负责治理已经形成的热点。两者配合，才能形成更完整的负载治理链路。

最后是增强 Shadow Load 可观测性。后续可以把影子负载、Pod 估算值、风险系数生效情况暴露为调度器内部指标，让用户看到节点为什么降分、哪些 Pod 正在贡献影子负载、某次发布为什么没有继续落到热点节点。这对运维排障和客户信任都很重要。

## 总结

负载感知调度解决的是 Kubernetes 只看资源声明导致的调度偏差。真实集群里，request 和运行时消耗之间经常存在偏差；批量发布和重部署时，监控指标延迟又会让调度器短暂低估刚承接新 Pod 的节点压力。

Volcano usage 插件把真实指标、节点阈值、节点打分、影子负载、资源预估和回滚一致性放进同一套调度路径。外部指标提供节点真实负载，调度事件补齐监控尚未上报的新负载，资源预估让不同 QoS 类型的 Pod 都能参与节点评分，快照回退保证 Gang 调度失败后不污染下一轮决策。

对用户来说，这套方案的价值很直接：在批量新建、滚动重部署、扩缩容和 Gang 调度等场景下，减少新热点形成的概率；在业务资源声明不够准确时，可以通过参数把业务经验沉淀到调度策略里；在和其他调度方案对比时，也能体现 Volcano 面向批处理和 AI 场景的一致性、可解释性和演进空间。
