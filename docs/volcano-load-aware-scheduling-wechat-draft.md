# 让可观测数据进入调度闭环：Volcano 负载感知调度增强实践

在云原生调度系统里，“资源是否足够”并不总是等价于“这个节点现在适不适合继续放任务”。

传统调度逻辑更关注资源声明，例如 Pod 的 request、limit，以及节点的 allocatable。这个模型稳定、可解释，也很适合做准入判断。但在实际集群里，运行时负载往往比静态资源声明更复杂：有的任务 CPU request 不高，但运行时会突然拉高；有的节点资源分配率看起来不高，但已经被存量任务压到较高水位；还有一些 BestEffort 或弹性任务，本身就很难仅靠 request 准确描述真实消耗。

这也是 Volcano usage 插件要解决的问题：让调度器不只看到“资源账本”，也能看到节点运行时的 CPU、内存负载，并把这些可观测数据纳入调度决策。

在上一篇负载感知调度文章中，重点解决的是“调度器如何看见节点真实负载”的问题。它通过 usage 插件接入 Prometheus、Custom Metrics API、Elasticsearch 等外部指标源，在 Predicate 阶段过滤超过 CPU 或内存阈值的节点，并在 NodeOrder 阶段优先选择低负载节点。由此，Volcano 的调度决策不再只依赖 Pod request、limit 和节点 allocatable，而是开始把运行时负载纳入调度排序。

不过，上一篇文章中的能力仍有一个边界：外部指标源反映的是已经被采集、聚合和拉取到调度器缓存中的负载，而不是调度事件发生瞬间的负载。一个 Pod 刚刚被分配到节点上时，调度器已经知道这次分配，但监控系统往往还没来得及把它体现在节点 CPU、内存曲线上。在连续调度、高并发提交或短时间负载突增的场景下，这个时间差会影响后续 Pod 的调度判断。

本文关注的正是这个问题：在已有 usage-based scheduling 的基础上，进一步补齐监控延迟窗口内的负载感知能力，让调度器不仅使用外部可观测数据，也把自己刚刚做出的调度决策纳入当前负载视图。

## 从“按资源调度”到“按负载调度”

Volcano 作为面向批处理、高性能计算、AI 等场景的 Kubernetes 原生调度系统，除了常规的资源维度，也支持通过 usage 插件引入节点运行时指标。为了理解这次增强的价值，我们先快速回顾 usage-based scheduling 在调度链路中的位置。

原有 usage-based scheduling 的基本思路可以概括为两步：

第一，在 Predicate 阶段过滤掉负载超过阈值的节点。例如当节点 CPU 或内存使用率超过用户配置的阈值时，不再向该节点调度新的 Pod。

第二，在 NodeOrder 阶段根据节点负载进行打分。负载越低的节点，获得越高的分数，从而在调度排序中更容易被选中。

这个机制让调度器从“只看资源声明”走向“同时参考运行时负载”。在很多场景下，它可以有效缓解节点间真实负载不均的问题，也为本文要讨论的增强能力提供了基础。

因此，本文不是对前一篇文章的重复介绍，而是在它的基础上继续往前走一步：前文解决“调度器能不能看见负载”，本文解决“调度器看见的负载是否足够接近当下真实状态”。

不过，问题并没有到此结束。

节点负载通常来自 Prometheus、Custom Metrics API 或 Elasticsearch 等外部指标源。调度器周期性拉取这些指标，再把结果缓存到调度周期中使用。这里天然存在一个时间窗口：一个 Pod 刚刚被调度到节点上时，它已经会影响节点未来负载，但它的实际 CPU、内存使用还没有被监控系统采集、聚合并反馈回来。

如果调度器只依赖外部指标源返回的当前值，就可能出现这样的情况：

节点 A 在指标里看起来很空闲；调度器连续把多个 Pod 分配到节点 A；等下一轮指标更新时，节点 A 的负载已经明显升高，甚至超过预期。

这个窗口就是负载感知调度里的“监控盲区”。

## 这次增强解决的核心问题

这次负载感知调度增强的目标，不是推翻原有 usage 插件，而是补齐它在监控延迟窗口内的判断能力。

核心思路是：调度器不能只相信已经被监控系统观测到的真实负载，还要把“已经调度成功、但尚未进入监控指标”的 Pod 也算进去。

因此，我们在 usage 插件中引入了一个 session 级别的 Shadow Load 机制。

所谓 Shadow Load，可以理解为调度器内部维护的一份“影子负载”。它不来自 Prometheus，而来自当前调度周期内的调度行为。当一个 Pod 被分配到某个节点后，usage 插件会根据该 Pod 的资源声明和当前节点压力，为它估算一份 CPU、内存消耗，并暂时累加到该节点的影子负载中。

最终用于打分的节点负载，就不再只是：

```
real_load
```

而是：

```
real_load + shadow_load
```

这样一来，即使外部监控指标还没有更新，调度器也能在当前 session 内意识到：“这个节点刚刚已经被放了一批新任务，后续 Pod 不应该继续盲目涌入。”

## Shadow Load 如何工作

这次实现中，Shadow Load 的关键组件是 `ShadowLoadCache`。它按节点维护两类估算值：CPU 估算值和内存估算值。同时，它还为每个 Pod 保存一份估算快照。

为什么需要快照？

因为调度并不总是只发生“分配成功”这一种路径。调度过程中可能发生回滚，Pod 也可能在不同节点之间经历状态变化。如果 Allocate 时根据当时的公式和节点压力加了一份估算值，Deallocate 时再重新计算一次，就可能因为风险系数、配置或节点状态变化导致加减不一致。

因此，这次增强采用了快照式扣减：Allocate 时记录“当时加了多少”，Deallocate 时就严格扣回“当时那一份”。这能保证影子负载在调度回滚时保持一致，不会因为重复计算出现残留或误差累积。

除了调度事件中的增减，usage 插件在 session 打开时还会做一次 warm-up。它会扫描已经分配到节点上的任务，把处在监控盲区内的 Pod 加入 Shadow Load。比如刚刚 Bound、Binding、Allocated 的任务，或者已经 Running 但启动时间还很短、尚未被 metrics interval 覆盖的任务。

这一步很重要。否则如果调度器重开一个 session，只从空的 Shadow Load 开始，就会丢掉上一轮刚刚调度但还没被监控系统看见的任务。

## 如何估算一个 Pod 的负载

Shadow Load 的另一个关键点是估算模型。

对于 Guaranteed 或 Burstable 类型的 Pod，估算公式可以简化理解为：

```
pod_estimate = (request * request_ratio + (limit - request) * burst_ratio) * applied_risk_factor
```

其中，`request_ratio` 表示 request 对估算值的贡献，`burst_ratio` 表示 request 到 limit 之间这段 burst 空间的贡献。估算结果会被限制在合理范围内：不会低于 0，也不会超过有效 limit；如果某个资源没有配置 limit，则以 request 作为有效上限。

对于 BestEffort Pod，由于它没有明确的 request/limit，插件使用配置项提供固定估算值，例如默认 CPU 为 `250m`，内存为 `200Mi`。

更进一步，估算模型还引入了风险系数 `risk_factor`。

当节点已经处于较高复合负载时，新 Pod 带来的风险通常更高。此时插件会根据 CPU、内存的组合利用率判断是否超过 `risk_threshold`。一旦超过阈值，估算值会乘以 `risk_factor`，让调度器在高压节点上更加保守。

这个模型的好处是，它不是简单地把所有新 Pod 都按最大 limit 计算，也不是完全相信 request，而是在 request、burst 空间、节点当前压力之间做一个可配置的折中。

[可补充：这里非常适合加入竞品分析。可以比较几类方案：只按 request 调度、只按实时指标调度、固定安全冗余、基于调度事件维护影子负载。重点说明本方案为什么在实时性、保守性和可配置性之间更平衡。]

## Predicate 与 NodeOrder 的分工

在调度流程上，这次增强没有把所有逻辑都塞进过滤阶段，而是保持了 Predicate 与 NodeOrder 的分工。

Predicate 阶段仍然负责硬性判断：如果真实指标显示节点 CPU 或内存已经超过阈值，就过滤该节点。这一部分依赖真实观测指标，避免把已经明显过载的节点继续纳入候选。

NodeOrder 阶段负责排序：它使用真实负载和 Shadow Load 计算复合利用率，再把复合利用率转换为节点得分。复合利用率越低，节点得分越高；CPU 和内存的权重可以通过配置调整；usage 插件本身的权重也可以参与最终调度分数。

这样设计的意义在于：硬阈值仍然基于真实观测，避免估算值造成过度过滤；排序阶段则引入影子负载，让调度器在候选节点之间做更接近实时状态的选择。

换句话说，Predicate 负责“不该去哪里”，NodeOrder 负责“更应该去哪里”。

## 配置方式

这次增强延续 usage 插件已有配置方式，同时新增 estimator 相关参数：

```yaml
actions: "enqueue, allocate, backfill"
tiers:
  - plugins:
      - name: usage
        enablePredicate: false
        arguments:
          usage.weight: 5
          cpu.weight: 1
          memory.weight: 1
          thresholds:
            cpu: 80
            mem: 80
          estimator:
            request_ratio: 0.7
            burst_ratio: 0
            risk_threshold: 0.6
            risk_factor: 1.2
            be_cpu: 250m
            be_memory: 200Mi
metrics:
  type: prometheus
  address: http://prometheus:9090
  interval: 30s
```

其中，`metrics.interval` 不只影响指标采集频率，也被 usage 插件用于判断监控延迟窗口。对于刚启动不久的 Running Pod，如果它的运行时间还没有覆盖这个 interval，就会被认为仍处在监控盲区内。

在实际使用中，可以根据业务特征调整参数：

- 对负载比较稳定、request 较可信的任务，可以提高 `request_ratio`；
- 对突发负载明显的任务，可以适当提高 `burst_ratio`；
- 对高负载节点更敏感的场景，可以降低 `risk_threshold` 或提高 `risk_factor`；
- 对 BestEffort 任务较多的集群，需要结合历史数据校准 `be_cpu` 和 `be_memory`。

## 调度效果如何验证

针对这类调度增强，单纯看代码逻辑是不够的，更重要的是验证它在不同场景下是否真的改变了调度结果。

建议从几个维度观察：

第一，节点分散度。连续提交多个 Pod 或多个 Job 时，增强前后 Pod 在节点上的分布是否更均衡。

第二，热点规避能力。当某个节点在短时间内已经被分配较多新任务时，后续任务是否会倾向于调度到其他节点。

第三，高负载保护能力。当节点真实负载接近阈值时，风险系数是否能让打分更保守，减少继续向该节点分配任务的概率。

第四，回滚一致性。当调度过程中发生 Deallocate 或调度回滚时，Shadow Load 是否能准确扣减，不影响后续调度决策。

第五，BestEffort 任务表现。在没有 request/limit 的情况下，固定估算值是否能有效避免 BestEffort Pod 被无限低估。

- 场景二：节点接近高负载场景，验证 `risk_factor` 是否让调度更保守；
- 场景三：BestEffort 或 burst 型任务场景，验证 estimator 参数是否能改善无 request 或 request 偏低带来的误判。

## 从可观测到可调度

这次负载感知调度增强，本质上是在 usage 插件中补齐了一个调度闭环：

可观测系统提供节点真实负载；调度器在此基础上维护监控盲区内的影子负载；调度决策再结合真实负载和影子负载，对后续 Pod 进行更合理的排序。

这让可观测数据不再只是用于事后分析或告警，而是真正进入调度路径，影响系统当下的资源分配。

对于 AI、批处理、Agent 等负载波动更明显的场景，这类能力尤其重要。因为这些场景里的任务形态往往更加动态：有的任务启动阶段消耗高，有的任务运行时呈现 burst，有的任务 request 与真实负载之间存在偏差。调度器如果只看静态资源声明，就很难捕捉这些变化；如果只看外部实时指标，又会受到采集延迟限制。

Shadow Load 的价值就在于，它把调度器已经知道、但监控系统尚未反映的信息补进来了。

## 后续展望

目前这次增强主要围绕 CPU 和内存两个维度展开，解决的是监控延迟窗口内的负载估算与节点打分问题。后续还可以继续从几个方向演进：

一是引入更多资源维度，例如 GPU、NPU、网络、磁盘 IO 等，让异构资源场景下的负载感知更完整。

二是结合历史数据优化估算模型。当前 estimator 提供了可配置参数，后续可以根据任务画像、队列画像或历史运行数据进一步自动校准。

三是与重调度能力联动。调度阶段可以避免新增热点，重调度阶段则可以处理已经形成的热点，两者结合才能形成更完整的负载治理能力。

四是增强可观测性。Shadow Load 本身也可以暴露为调度器内部指标，帮助用户理解一次调度为什么选择某个节点，以及某个节点为什么在短时间内分数下降。

从这个角度看，负载感知调度不仅是一个 scheduler plugin 的增强，也是 Volcano 面向动态工作负载持续演进的一部分：让系统看得见负载，也能基于负载做出更及时的调度决策。
